# Context: 000023_context-over-memory

## Current State Analysis

The repository already has all the per-agent instruction surfaces in place; none of them currently carry any guidance about where agents should persist learnings.

- `CLAUDE.md` (3 lines) is a thin shim that imports `AGENTS.md` via `@AGENTS.md`. Claude Code resolves `@-imports` at session start.
- `AGENTS.md` (5 lines) contains only the existing tessl-managed block:
  - Line 3: `# Agent Rules <!-- tessl-managed -->`
  - Line 5: `@.tessl/RULES.md follow the [instructions](.tessl/RULES.md)`
  - The `<!-- tessl-managed -->` marker indicates `tessl install` may regenerate or replace text inside this block. The block ends at line 5; everything appended below it is outside tessl's purview.
- `.codex` is a zero-byte marker file at the repository root. Codex loads `AGENTS.md` from the repository root natively — the marker file is unused for instruction content.
- `.bob/rules/rules.md` contains hand-written Bob guidance about documentation queries. Bob loads all `.md` files in `.bob/rules/` automatically. Per user confirmation, Bob also reads `AGENTS.md` at session start, so we do not need a Bob-side change.
- `.tessl/RULES.md` is regenerated by `tessl install` (its own body says so) — unsafe to edit by hand.

The shared knowledge store this plan redirects writes into already exists:

- `.spektacular/knowledge/` — the project-scoped knowledge store configured under spec 000020. Currently empty (`go run . knowledge search "agent instructions memory context"` returns zero hits).
- `go run . knowledge {sources,search,read,write}` — CLI surface for the store, with `write` enforcing the propose-then-confirm guard.
- `spek-knowledge` skill — agent-facing wrapper around the CLI, shipped under spec 000022. Listed in the available-skills surface for Claude, with equivalents installed for Codex and Bob by `internal/agent/{codex,bob}.go`.

The Spektacular CLI's own agent-install code (`internal/agent/{claude,codex,bob}.go`) only manages skills and command wrappers — none of it touches `CLAUDE.md`, `AGENTS.md`, `.codex`, or `.bob/rules/`. The edit in this plan therefore cannot collide with `go run . project init` or any other Spektacular subcommand.

## Per-Phase Technical Notes

### Phase 1.1: Add the embedded section template

**File changes**:

- `templates/agents/memory-context.md` — new file. The directory `templates/agents/` does not yet exist and must be created. `templates/embed.go` already declares `//go:embed all:*`, so any file added under `templates/` is picked up automatically — no Go change required for embedding.

**Draft body** (verbatim, or with light wording-only edits at implementation time):

```markdown
## Memory & Context

> Managed by `spektacular init` — edit `templates/agents/memory-context.md`
> in the Spektacular source, not this section in place. Hand edits will not
> survive the next init.

In this repository, do not persist anything to your per-user, per-machine
memory store. When you would normally write to it — a learning, convention,
gotcha, project fact, user preference, or anything else worth remembering
between sessions — route the write through the `spek-knowledge` skill
instead. The skill handles scope selection, search-before-write, and
propose-then-confirm.

Outside this repository, continue using your per-user memory store as normal.
```

**Why each load-bearing element is present**:

- The trigger phrase ("when you would normally write to [per-user memory]") is intentionally agent-neutral — it matches the framing Claude's auto-memory system uses (`~/.claude/projects/.../memory/`) and is general enough for Codex's and Bob's equivalents.
- The destination names the skill (`spek-knowledge`) only. The skill itself owns scope selection, search-before-write, propose-then-confirm, scratch-file staging, and the actual `{{command}} knowledge write` invocation (see `templates/skills/workflows/spek-knowledge/SKILL.md`). Naming any of those mechanics in `AGENTS.md` would duplicate the skill and create two places to update when the skill evolves.
- The in-repo scoping satisfies the spec's "Scoped to this repo" acceptance criterion.
- The "managed by `spektacular init`" callout tells humans why hand-edits will not survive and points them at the right place to make permanent changes — required because heading-based detection makes the section mutable on its surface.

**No `{{command}}` placeholders in the current body**: there are no CLI invocations to render. The template is still routed through `mustache.Render` on install (so the writer path is identical to `installWorkflowSkills`'s and so future edits can add `{{command}}` references without touching the writer), but for now the rendered output equals the template bytes.

**Complexity**: Low
**Token estimate**: ~1k (single new ~20-line markdown file)
**Agent strategy**: Single agent, sequential.

### Phase 1.2: Implement the idempotent AGENTS.md writer

**File changes**:

- `internal/agent/memory_context.go` — new file. Implements `installMemoryContextSection(projectPath string, cfg config.Config, out io.Writer) error`.

**Algorithm (canonical)**:

1. `tmplBytes, err := fs.ReadFile(sourceFS, "agents/memory-context.md")`. The package-level `sourceFS` is the same `templates.FS` (or test-injected fixture FS) that `installWorkflowSkills` uses, so no new variable is needed.
2. `rendered, err := mustache.Render(string(tmplBytes), map[string]string{"command": cfg.Command})`.
3. `agentsPath := filepath.Join(projectPath, "AGENTS.md")`.
4. Read existing contents: `existing, err := os.ReadFile(agentsPath)`. If `os.IsNotExist(err)`, treat `existing` as empty and proceed to the "create" branch.
5. Locate the section in `existing`:
   - Split on `"\n"` into a `[]string`.
   - Walk the slice; record the index of the first line whose trimmed-right form equals `"## Memory & Context"`.
   - If found, walk forward from `startIdx+1` looking for the next line whose trimmed-right form begins with `"# "` or `"## "` (any other H1/H2). The "end" index is that line's index, or `len(lines)` if no such line exists. The replaced span is `[startIdx, endIdx)`.
6. Compose the new file body:
   - **Missing-file branch**: body = `rendered + "\n"` if `rendered` does not already end with `\n`.
   - **Heading-found branch**: replace `lines[startIdx:endIdx]` with the lines from `rendered`. If the resulting slice has a non-blank line immediately before `startIdx` and the rendered section does not start with a blank line, insert a separator blank line. Same trailing rule.
   - **Heading-absent branch**: ensure `existing` ends with exactly one trailing newline followed by a blank line, then append `rendered`. Trim any extra trailing whitespace so we don't accumulate blank lines on repeated runs.
7. Write atomically: `os.WriteFile(agentsPath+".tmp", body, 0644)` then `os.Rename(agentsPath+".tmp", agentsPath)`. The temp file inherits the same directory so the rename is a same-filesystem atomic op.
8. Emit one line to `out`:
   - `"  AGENTS.md: created %s\n"` (missing-file branch).
   - `"  AGENTS.md: updated Memory & Context section in %s\n"` (heading-found branch).
   - `"  AGENTS.md: appended Memory & Context section to %s\n"` (heading-absent branch).
   The `%s` is the absolute `agentsPath`, matching `installWorkflowSkills`'s `"  Skill:    %s\n"`.

**Edge cases the implementation must handle**:

- The existing file ends without a trailing newline. The writer must add exactly one before appending in the heading-absent branch so the appended section does not glue onto the prior line.
- The heading appears at the very last line with no body below. `endIdx == len(lines)` correctly captures "to EOF".
- A user has added their own `## Foo` heading directly under `## Memory & Context`. `endIdx` lands on that line; the user's content is preserved.
- The repeated-run case: re-rendering and replacing produces the same bytes only if the template body itself is deterministic and the renderer is stable. `mustache.Render` is deterministic for these inputs.
- A user has hand-deleted the heading but left section body text below. The writer treats this as "absent" and appends a fresh section; the orphaned body text is left alone (the user can clean it up). This is acceptable per the "section is managed" contract.

**Complexity**: Medium (~80 lines of Go, plus careful boundary handling around line-array surgery).
**Token estimate**: ~6k (read existing agent files for pattern, write new file, run tests).
**Agent strategy**: Single agent, sequential. Coordinated with Phase 1.4 — the writer and its tests are most naturally authored together by the same agent.

### Phase 1.3: Wire the writer into each agent's Install

**File changes**:

- `internal/agent/claude.go` — `Install()` gains a single trailing call: `return installMemoryContextSection(projectPath, cfg, out)` after the existing `installWorkflowSkills` call (or wrap both in an explicit sequence with intermediate error handling).
- `internal/agent/codex.go` — same as `claude.go`.
- `internal/agent/bob.go` — append the call after the existing `installCommandWrappers` call. Bob's `Install()` already has multi-step sequencing, so the addition slots in naturally.

**Pattern** (illustrative, for `claudeAgent`):

```go
func (claudeAgent) Install(projectPath string, cfg config.Config, out io.Writer) error {
    if err := installWorkflowSkills(projectPath, ".claude/skills", cfg, out); err != nil {
        return err
    }
    return installMemoryContextSection(projectPath, cfg, out)
}
```

**Why this placement and not `project.Init` / `runInit`**: per the user's design answer for this plan, the writer is a shared helper called by each agent's `Install`. The action is the same for all three agents, but keeping it on the per-agent path (a) matches how Spektacular already factors per-agent installation (skills are also identical-per-agent but called from each `Install`), and (b) makes the agent-install surface (skills + commands + AGENTS.md section) cohesive and discoverable in one place per agent. Idempotency guarantees that re-running across agents on the same project is safe.

**Complexity**: Low (one-line addition × three files).
**Token estimate**: ~1k.
**Agent strategy**: Single agent, sequential. Cannot start until Phase 1.2 lands `installMemoryContextSection`.

### Phase 1.4: Add the idempotency unit tests

**File changes**:

- `internal/agent/memory_context_test.go` — new file. Table-driven `TestInstallMemoryContextSection` covering all cases listed under plan.md § Testing Approach §1.

**Test fixtures**: All under `t.TempDir()`. Each subtest pre-seeds `AGENTS.md` (or leaves it absent) with a hand-written string constant, runs `installMemoryContextSection`, and asserts the resulting file bytes against a hand-maintained `wantAgentsMD` string. Use `internal/agent/agent_test.go` for the existing fixture-FS test pattern — `sourceFS` can be substituted with a `fstest.MapFS` holding just `agents/memory-context.md` for predictable rendering.

**Idempotency subtest**:

```go
require.NoError(t, installMemoryContextSection(dir, cfg, io.Discard))
first, err := os.ReadFile(filepath.Join(dir, "AGENTS.md"))
require.NoError(t, err)

require.NoError(t, installMemoryContextSection(dir, cfg, io.Discard))
second, err := os.ReadFile(filepath.Join(dir, "AGENTS.md"))
require.NoError(t, err)

require.Equal(t, string(first), string(second), "second run must produce byte-identical output")
```

**Cross-agent idempotency subtest**: walk `Supported()`, call each agent's `Install` against the same temp dir, then assert (a) the resulting `AGENTS.md` has exactly one occurrence of `"## Memory & Context"` (via `strings.Count`), and (b) the byte content matches a hand-maintained expected string.

**Per memory `feedback_tests_need_independent_oracle`**: do *not* compute `wantAgentsMD` by calling the renderer or template engine — hand-maintain it as a verbatim string. Test maintenance cost is acceptable; the alternative is a test that cannot catch a renderer bug.

**Per memory `feedback_tests_own_their_filesystem`**: do not read or write anything outside `t.TempDir()`. Use `fstest.MapFS` for the template input.

**Per memory `feedback_no_redundant_assertions`**: the "exactly one section" check is the idempotency-specific assertion; the byte-equality check covers content correctness. Do not also assert on heading count when byte-equality already covers it.

**Complexity**: Medium (~150 lines including fixtures and string constants).
**Token estimate**: ~6k.
**Agent strategy**: Single agent, sequential with Phase 1.2.
