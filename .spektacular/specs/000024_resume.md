## Overview

Make it possible to resume an in-progress specification, plan, or implementation that was interrupted before it finished. Today, if a session ends partway through — a crash, a reboot, or simply closing the tool — there is no way to pick the work back up where it left off, so the half-finished effort is lost and has to be restarted from scratch. Anyone driving one of these workflows benefits: their progress survives interruptions and they can continue seamlessly instead of redoing completed steps.

## Requirements

- **Detect an in-progress workflow on start.** When a user starts a specification, plan, or implementation, the system must check whether a workflow of that same kind is already in progress.
- **Prompt to resume or start new.** If an in-progress workflow of that kind is found, the user must be asked whether to resume it or start a new one, rather than the system silently doing either.
- **Resume continues where it stopped.** When the user chooses to resume, the workflow must continue at the first step that was not yet completed; steps already completed must not be repeated.
- **Start-new leaves the existing one intact.** When the user chooses to start new, a new workflow must begin without discarding the in-progress one.
- **Preserve completed work on resume.** When a workflow is resumed, all content captured in steps that completed before the interruption must be intact and available.
- **Survive process termination.** Detection and resume must work even after the tool's process has fully exited (crash, reboot, or normal close), not only within a single running session.
- **No prompt when nothing is in progress.** If there is no in-progress workflow of the kind being started, the user must not be prompted and a new workflow must start as it does today.
- **Independent workflows per kind.** A specification, a plan, and an implementation may each be in progress independently; detection for one kind must not be affected by an in-progress workflow of another kind.

## Constraints

- **Must not lose or corrupt existing in-progress work.** Introducing resume must never discard or damage a workflow's already-captured state.

## Acceptance Criteria

- **Context persisted after each step.** After any step of a workflow completes, the workflow's persisted state reflects that step's captured content; if the process is terminated immediately after a step completes and the entry command is re-run in a fresh process, resuming re-reads that content and continues from the following step.
- **In-progress detected on entry.** Given an unfinished workflow, running the entry command reports that an in-progress workflow of that kind exists, so the user can be offered the choice to resume or start new.
- **Resume continues, no repeat.** Resuming an unfinished workflow whose first N steps are complete advances to step N+1; completed steps are not re-presented.
- **Completed work intact.** After resuming an interrupted workflow, reading its artifact shows previously captured content unchanged.
- **Start-new keeps the old one.** Choosing start-new yields a new workflow while the previous unfinished workflow still exists unchanged.
- **Survives process exit.** After the tool process is fully terminated, a brand-new process running the entry command still detects the in-progress workflow and can resume it.
- **No prompt when clean.** With no in-progress workflow of that kind, running the entry command reports none and begins a new workflow without offering resume.
- **Per-kind independence.** An in-progress workflow of one kind does not cause a resume report when the user starts a workflow of a different kind.

## Technical Approach

- **Per-step context persistence (already largely in place).** Workflow state (`current_step`, `completed_steps`, captured `data`) is persisted on every step transition, so an interrupted workflow can be reconstructed from the last persisted state after the process exits. Resume builds on this existing mechanism.
- **Detect instead of delete.** Today the entry commands (`spec new`, `plan new`, `implement new`) unconditionally remove the existing state file before starting. Change them to first load any existing state and check whether it represents an unfinished workflow of the same kind (current step is not the terminal/`finished` step).
- **Record the workflow kind in state.** Persisted state currently has no field identifying which kind of workflow it belongs to (only `data.name`). Add a workflow-kind marker (spec / plan / implement) so detection can be scoped per kind.
- **Per-kind state storage.** Move from the single shared `.spektacular/state.json` to per-kind state so an in-progress spec and an in-progress plan can coexist without clobbering each other (separate state files per kind, or a single file keyed by kind — left to the planner). This is required to honour per-kind independence.
- **Report, don't auto-act.** When an unfinished workflow is detected, the entry command surfaces it in its JSON output with enough identity (kind + name + current step) for the driving agent to ask the user resume-vs-start-new — the CLI itself is non-interactive. Resume continues from `current_step`; start-new begins a fresh workflow while leaving the existing state intact.
- **Known risk / migration.** Moving from a single shared `state.json` to per-kind state is the main structural change; existing single-file installs must migrate cleanly so in-progress state is neither lost nor corrupted.

## Success Metrics

- **Zero restart-from-scratch after interruption.** After a crash, reboot, or closed session mid-workflow, users resume and complete rather than starting over — no lost in-progress work reported.
- **No accidental clobbering.** No reports of an in-progress workflow being silently destroyed by re-running an entry command.

## Non-Goals

- **Not a general undo/history.** This does not provide rewind to arbitrary earlier steps or version history of a workflow — only continuing an unfinished one from where it stopped.
- **No multi-user/concurrent editing.** Coordinating two people resuming the same workflow simultaneously is out of scope.
- **No cross-machine sync.** Resuming on a different machine than where the workflow was interrupted is out of scope.
