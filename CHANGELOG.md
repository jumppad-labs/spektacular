## 000023_context-over-memory

Spektacular now redirects what coding agents would normally save into their own per-user, per-machine memory store into the project's shared knowledge base instead. When you work in a Spektacular-initialised repo with Claude, Codex, or Bob and the agent identifies a convention, gotcha, project fact, or preference worth remembering, it proposes a write to the project knowledge store via the `spek-knowledge` skill — you review the entry, choose the scope, and confirm before anything is persisted — so the captured knowledge ends up in git and travels with the project for every collaborator. Outside this repo, the same agents keep using their per-user memory as normal. The redirect is installed as a managed "Memory & Context" section in `AGENTS.md` by `spektacular init`; re-running init for the same agent or a different one updates that section in place without duplicating, and it sits alongside (not inside) any existing `tessl install` block. Existing projects pick this up by re-running `spektacular init <agent>` once.

## 000022_spek-knowledge-skill

Spektacular now ships a fourth workflow skill — `spek-knowledge` — that lets you read, contribute to, and update the project's knowledge base in any session without starting a formal spec or plan flow. Ask in natural language ("what do we know about X?", "remember Y", "update what we have on Z") and the agent runs the matching `spektacular knowledge` command, shows search results with their source scope labelled, and — for any write — presents the proposed scope, path, and body and waits for your explicit confirmation before persisting. The skill installs alongside `spek-new`, `spek-plan`, and `spek-implement` for every supported agent (claude, codex, bob); existing projects pick it up by re-running `spektacular init <agent>`.

## 000021_file-writing

Writing a spec or plan file through Spektacular no longer goes through standard input. `spektacular spec file write` and `spektacular plan file write` now take a required `--from <path>` flag pointing at a source file on disk, and the contents arrive in the store byte-for-byte — backticks, dollar signs, quotes, and newlines all survive intact, where the previous shell-piped form would mangle them. Pointing `--from` at a missing path fails cleanly with the offending path in the error message, and the source file is left untouched after a successful write. The workflow skill and step prompts shipped with Spektacular have been updated to drive the new form (stage under `.spektacular/tmp/`, write with `--from`, `rm` the scratch file), and a regression test guards the instruction surface so the old stdin form cannot creep back in. The stdin path has been removed outright with no compatibility shim, so any external script that piped content into `file write` must switch to `--from <path>`.

## 000020_context

Spektacular now draws on a multi-source knowledge layer when planning. Instead of one fixed knowledge folder, a project can configure several knowledge sources at distinct scopes — `project`, `team`, `global`, and so on — each independently located and keyword-searchable, so teams can share architectural notes, conventions, and past learnings while keeping private ones private. A new `spektacular knowledge` command lets agents search every source with one query (results tagged by where they came from), read a full entry, list everything available, write a new entry into a chosen scope, and inspect the configured sources; an unreachable source fails loudly rather than silently returning a partial answer. Configuration is now provider-based: `spec`, `plan`, and `knowledge` each name a provider and a config block, which also makes the spec and plan output directories configurable for the first time. Projects with no `knowledge` configuration keep working unchanged via a synthesised default source. The `config.yaml` format has changed — `spec`, `plan`, and `knowledge` each gain a `provider` field and a provider-specific `config` block; `spec.id_method` stays a sibling of `provider` since identifier generation is independent of the storage backend. Update existing config files to the new shape, or remove the stale section to fall back to defaults.

## spec-id-prefix-method

Spec creation now uses CLI-owned identifier prefixes. By default, `spec new` creates timestamp-prefixed spec names, accepts an optional `id` for external systems, and can be configured with `spec.id_method: timestamp`, `counter`, or `external`. Callers should use the returned `spec_name` and `spec_path` because requested names and ids are normalized before workflow state and files are created.

## 19_codex_integration

Spektacular now ships first-class support for OpenAI Codex alongside Claude Code and Bob. `spektacular init codex` installs the three workflow entry points — spec, plan, and implement — as native SKILL.md files, so Codex users can invoke `$spek-new`, `$spek-plan`, and `$spek-implement` out of the box. As part of this release the workflow entry points become the canonical artefact type across every supported agent: slash commands for Claude and Bob are now thin wrappers that simply invoke the matching skill, and workflow instructions live in one place rather than being copied per-agent. Existing Claude and Bob users should re-run `spektacular init <agent>` once to pick up the new layout.

## 18_release_workflow

Spektacular now has a complete automated release pipeline powered by Dagger and GitHub Actions. Every push to any branch builds cross-platform binaries (darwin/linux on amd64/arm64) with automatic macOS code signing and notarization. Merging to `main` with a `release/patch|minor|major` PR label automatically publishes a new release to GitHub Releases, updates the Homebrew formula in `jumppad-labs/homebrew-repo`, and pushes packages to GemFury. Users can now install spektacular via `brew install jumppad-labs/repo/spektacular` or download signed binaries directly from GitHub Releases, eliminating the need to clone and build from source.
